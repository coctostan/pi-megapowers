- The done phase's `blocking: true` flag prevents writing to CHANGELOG.md, docs/, or any non-.megapowers/ path during done-phase actions — done-phase wrap-up actions that need to write to the project (changelog, feature docs) must write to `.megapowers/artifacts/` paths instead, or the workflow config needs to change the done phase to not be blocking.
- Pi loads extension modules at startup and caches them — code changes made during a session (e.g., by subagents) don't take effect until pi restarts; this means implementing changes that affect the current session's tools is a bootstrapping problem.
- When issue 065 changed `handleTaskDone` to support `phase === "done"`, the running session couldn't use the new behavior immediately because the old module was cached in memory — design tools and workflows to be robust to this kind of version mismatch.
- The `completedDoneActions` field not being in `KNOWN_KEYS` in old code means that even if a new state field is added, old readers will silently drop it — always add new fields to KNOWN_KEYS in the same commit as the interface change.
- Done-phase actions should explicitly handle the case where the write policy blocks them — either the done phase should not be `blocking: true`, or done-phase actions should be designed around the `.megapowers/` write-allowance.
- The `squash-task-changes` action in the old done-phase system predates `vcs-wrap-up` — when migrating action names, old in-flight `doneActions` arrays in state.json won't be updated automatically; consider a migration step or graceful handling of unknown action keys.
