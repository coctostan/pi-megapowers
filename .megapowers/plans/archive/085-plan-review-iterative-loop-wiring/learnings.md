# Learnings — 085-plan-review-iterative-loop-wiring

- **Template variables that are structurally impossible to populate are silent UX landmines.** `review-plan.md` used `{{plan_content}}` but `plan.md` is only generated on approval — it can't exist when review mode is active. `interpolatePrompt` leaves unresolved keys as literal text (`return vars[key] ?? match`), so the reviewer would see `{{plan_content}}` verbatim. Always verify that every `{{VAR}}` in a prompt template can actually be populated at the time that template is loaded; if it can't, replace it with a read-from-disk instruction instead.

- **Internal dev notes in return values leak into production.** The revise verdict message contained `"(see Task 18 wiring)"` — a plan reference with zero meaning to LLMs or users. Text returned from tool handlers is LLM-visible; treat it like user-facing copy, not inline comments. Review tool messages the same way you review error strings.

- **`{ error: string }` branches need explicit handling at every call site.** When `readPlanTask` can return `null | EntityDoc<T> | { error: string }`, callers that only check `if (existing)` silently fall through the error case. Here, a corrupt task file was treated as "not found", causing the create path to overwrite it. Every discriminated union return type needs a dedicated branch for each variant — exhaustiveness checking (or a lint rule) would have caught this at compile time.

- **Tool schema descriptions are part of the LLM UX.** The `megapowers_signal` description advertised `review_approve` as an active action and `review→plan` as a valid `phase_back` target long after both were removed. LLMs rely heavily on the description field to decide which tools to try; stale descriptions cause ghost-invocations of deprecated actions. Update descriptions atomically with the underlying behavior change.

- **Redundant state mutations obscure intent.** `handleApproveVerdict` explicitly set `planMode: null as any` before calling `transition()`, which already resets `planMode` when leaving the plan phase. The redundant write plus the `as any` cast made the code look like it was working around a type error rather than expressing intent. When `transition()` owns the reset logic, callers shouldn't repeat it — the duplication signals a missing abstraction or a misunderstanding of ownership.

- **Code-review phase is a good place to catch prompt template bugs that tests can't.** Tests verified that the `review-plan.md` template was loaded (checked its opening line), but not that its variables were resolvable. The `{{plan_content}}` bug was invisible to the test suite. A companion "template variables resolv­ability" test — one that loads the template with a real store snapshot and asserts no `{{...}}` literals remain in the output — would catch this class of bug automatically.

- **Iteration caps on write-then-check flows need careful sequencing.** `handleReviseVerdict` checks `planIteration >= MAX_PLAN_ITERATIONS` after `writePlanReview` and `updateTaskStatuses` have already run. On the cap-exceeded path, the review artifact and task-status side effects are committed but the state is not advanced — leaving the system in a partially mutated but recoverable state. For "hard stop" guards like this, the check should come before side effects if possible; if side effects are intentional (audit trail), the partial state should be documented.
