# Learnings â€” 082: Reviewer-Authored Revise Instructions Handoff

- **File existence gates before state mutations are the cleanest architectural pattern.** The `handlePlanReview` gate (`existsSync` check before any `writePlanReview`/`writeState` call) keeps the function clean: failure is a pure early return with no partial side effects. This should be the default position for any validation that guards irreversible state changes.

- **`?? ""` is the right fallback for optional template variables.** The choice of `content ?? ""` over `if (content !== null) { vars.x = content }` is important: the former always sets the variable (replacing the template token with an empty string), while the latter leaves the token in the output when the file is absent. Never leave a template variable un-replaced in production output â€” always provide a fallback, even if empty.

- **Template variable injection should be co-located with the template selection.** Placing the `plan_iteration`/`revise_instructions` injection block immediately before the `PLAN_MODE_TEMPLATES` selection in `buildInjectedPrompt` makes the relationship obvious: "these vars are computed here because they're used in the template below." Scattering injection across multiple locations would make the data flow harder to trace.

- **Spy-based tests are more direct than output-inspection for "did not call" assertions.** The AC3 test (draft mode doesn't read revise-instructions files) patches `store.readPlanFile` and observes call arguments directly. This is strictly more robust than checking the template output â€” it doesn't depend on the template text or how unresolved tokens render. Use spy patterns when the invariant is "X was never called," not "output doesn't contain Y."

- **Updating existing tests alongside a breaking change is part of the implementation, not cleanup.** Adding the `existsSync` gate to `handlePlanReview` broke four existing tests (they now fail the gate because they don't create the fixture file). Fixing those tests as part of Task 5 â€” not deferred to a separate cleanup task â€” keeps the test suite green throughout the implementation and prevents false-negative test runs.

- **The `store?` optional guard pattern needs a convention.** When a `buildInjectedPrompt` caller omits the `store` argument and the template has a `{{revise_instructions}}` placeholder, the token passes through literally (because `vars.revise_instructions` is never set and `interpolatePrompt` falls back to the raw match). In practice the store is always provided, but the pattern `if (state.planMode === "revise" && store)` is subtly inconsistent â€” it makes the variable conditional on store presence when it should always be set in revise mode. Future convention: use `store?.readPlanFile(...)` with `?? ""` so the variable is always populated regardless of store availability.

- **File-existence gating at the tool boundary enforces workflow discipline without LLM instruction alone.** Prompting the reviewer to write the revise-instructions file before calling the tool works most of the time, but is not reliable on its own. The gate in `handlePlanReview` makes the constraint structural and deterministic â€” the tool simply refuses to accept a revise verdict without the file, so the reviewer *must* comply before the workflow can advance.
